# Regression Test Plan (Onboarding -> Access -> Plans -> Referrals -> Stat2)

Цель: пройти полный путь пользователя с нуля и убедиться, что монетизация, доступы и рефералка работают без регрессий.

## 0) Подготовка

- Нужны 2 Telegram-аккаунта:
  - `User A` (приглашающий)
  - `User B` (приглашенный)
- Бот должен быть задеплоен с актуальным кодом.
- В БД должны быть применены миграции (включая referrals).

## 1) Сброс состояния (стартовать всегда с этого)

### 1.1 Очистка `User A`

- В админ-чате выполнить:
  - `/removeuser <telegramChatId_UserA>`
- Ожидаемо:
  - бот подтверждает удаление пользователя и связанных данных.

### 1.2 Очистка `User B`

- В админ-чате выполнить:
  - `/removeuser <telegramChatId_UserB>`
- Ожидаемо:
  - бот подтверждает удаление пользователя и связанных данных.

## 2) Onboarding smoke

### 2.1 `/start` для `User A`

- Открыть бота как `User A`, вызвать `/start`.
- Ожидаемо:
  - показывается новый стартовый текст;
  - есть только одна кнопка: `Получить доступ`.

### 2.2 Меню команд

- Проверить список команд в Telegram-меню.
- Ожидаемо:
  - есть: `start`, `companies`, `news`, `referrals`;
  - нет: `plans`, `jobsearch`, `applications`, `hireagent`, `profile`, `about`.

## 3) Access / Jobsearch flow

### 3.1 Открытие jobsearch

- Нажать `Получить доступ` в `/start` (или вызвать `/jobsearch` вручную).
- Ожидаемо:
  - открывается mini app `seeker-jobs`;
  - нет постоянного блока `Subscription status`.

### 3.2 Required channels gate

- Пользователь НЕ подписан на required channels.
- Попробовать открыть детали вакансии.
- Ожидаемо:
  - показывается `subscribe_required` flow с каналами и кнопкой проверки.

### 3.3 После подписки на required channels

- Подписаться на required channels.
- Нажать `I subscribed - check again`.
- Ожидаемо:
  - доступ к деталям вакансии разрешается;
  - пользователь проходит дальше без ошибки.

## 4) Paywall / Plans flow

### 4.1 Исчерпание лимита

- Дойти до лимита открытий (или временно уменьшить конфиг).
- Открыть следующую вакансию.
- Ожидаемо:
  - появляется `payment_required` модалка;
  - в модалке есть:
    - тарифы для апгрейда,
    - кнопка `Пригласить друга (+бонус)`.

### 4.2 Flow через `/plans`

- В `User A` вызвать `/plans`.
- Нажать `Pricing`.
- Ожидаемо:
  - открывается pricing TMA-страница, не `seeker-jobs`;
  - отображаются тарифы и описания;
  - кнопка покупки открывает Telegram оплату Stars.

### 4.3 Оплата тарифа (опционально в проде)

- Совершить тестовую оплату тарифа.
- Ожидаемо:
  - подписка активируется;
  - лимиты соответствуют плану;
  - `successful_payment` не создает дубль при повторном событии.

## 5) Referral flow (основной)

### 5.1 Создание реферальной ссылки (`User A`)

- Вызвать `/referrals`.
- Ожидаемо:
  - бот показывает бонус за приглашение;
  - есть персональная ссылка вида `?start=<chatId_UserA>`;
  - есть кнопка `Поделиться ссылкой`.

### 5.2 Приглашение (`User B`)

- `User B` открывает бота по ссылке `User A`.
- Ожидаемо:
  - создается связь `A -> B`;
  - `User A` получает уведомление о начислении бонуса.

### 5.3 Проверка идемпотентности

- `User B` повторно вызывает `/start`.
- Ожидаемо:
  - второй бонус `User A` не начисляется.

### 5.4 Self-referral guard

- Открыть `/start <own_chat_id>` тем же пользователем.
- Ожидаемо:
  - бонус не начисляется, ошибок нет.

### 5.5 Invalid payload guard

- Вызвать `/start abc` или другой нечисловой payload.
- Ожидаемо:
  - падений нет;
  - referral side-effects отсутствуют.

## 6) Config-driven checks

### 6.1 Referral bonus config

- Изменить `Configs.ReferralBonusOpens` (например, 10 -> 15).
- Пригласить нового пользователя (новая пара referrer/referred).
- Ожидаемо:
  - начисляется новый размер бонуса (15), старые начисления не пересчитываются.

### 6.2 Free/bonus limits config

- Проверить, что при изменении `FreeJobOpensMonthlyLimit` и `ChannelSubscribeBonusOpens` поведение меняется без кода.
- Ожидаемо:
  - лимиты/бонусы подхватываются корректно.

## 7) Минимальный чек-лист “Go / No-Go”

- [ ] `/start` и стартовая кнопка работают
- [ ] `jobsearch` открывается без постоянного subscription блока
- [ ] required channels gate работает
- [ ] paywall показывает upgrade + invite friend
- [ ] `/plans` -> `Pricing` открывает корректную pricing-страницу
- [ ] `/referrals` выдает ссылку и share-кнопку
- [ ] referral бонус начисляется один раз на приглашенного
- [ ] self-referral и invalid payload не ломают flow
- [ ] hidden `/stat2` открывает web-dashboard с графиком и корректными метриками

## 8) Hidden admin command `/stat2`

### 8.1 Доступность команды

- Под обычным пользователем вызвать `/stat2`.
- Ожидаемо:
  - бот отвечает `Unauthorized.` (или команда не дает данных), метрики не раскрываются.
- Под админом (из `BOT_ADMIN_TELEGRAM_IDS`) вызвать `/stat2` в group/supergroup.
- Ожидаемо:
  - бот отвечает, что команда работает только в private-чате.
- Под админом вызвать `/stat2` в private-чате.
- Ожидаемо:
  - приходит кнопка открытия `Stat2` dashboard;
  - открывается страница `/app/stat2` (не текстовый отчет в чате).

### 8.2 Периоды и переключение

- В private-чате админа вызвать:
  - `/stat2`
  - `/stat2 1`
  - `/stat2 7`
  - `/stat2 30`
- Ожидаемо:
  - без аргумента кнопка открывает `/app/stat2?period=7`;
  - с аргументом период передается в URL (`period=1/7/30`).
- На странице `Stat2` переключать период через селектор (1/7/30/90) и `Reload`.
- Ожидаемо:
  - график и totals обновляются на выбранный период без ошибок.

### 8.3 Проверка метрик (минимальная валидация)

- Сверить значения из `/stat2` с БД за тот же период:
  - `Users joined` = количество `Users` по `DateJoined >= periodStart`;
  - `Users joined by invite` = количество `Referrals` по `ReferredAt >= periodStart`;
  - `Payments` = количество `TelegramPayments` по `PaidAt >= periodStart`;
  - `Required channel users` = уникальные пользователи `RequiredChannelUsers` по `(Date, UserId)` в периоде.
- Ожидаемо:
  - значения совпадают (допустима разница только из-за новых событий во время проверки).

### 8.4 Негативные сценарии

- Вызвать `/stat2 abc` и `/stat2 -10`.
- Ожидаемо:
  - команда не падает;
  - ссылка открывает dashboard с безопасным fallback-периодом (дефолтный 7).
- Временно отключить админ-режим (пустой `BOT_ADMIN_TELEGRAM_IDS`) и вызвать `/stat2`.
- Ожидаемо:
  - команда отвечает, что отключена в окружении.
